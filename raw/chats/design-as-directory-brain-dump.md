## Owner Message

> Another speech to text brain dump: - [ ] Rethinking the design type. One pattern that I’ve noticed is that when I’m using the design type document, I’m often mixing concerns. My current design document are a combination of target plus faced scoping. The target specter finds the future state but often the future state gets really complicated, especially for a mature software product. The phased scoping usually limits itself to just the first phase, and it also gives the opportunity to get more specific with certain implementation details. This is especially true when modifying an existing system and the need to design within the boundaries of existing abstractions or interfaces that should not be touched as part of the design. What I don’t like about it is that the design page no longer becomes a standalone requirements document and instead is that or tries to be that plus a transient implementation plan. I don’t mean implementation plan about the level of technical detail because depending on the design that’s necessary. What I mean is do this first and then at the end of this first step here’s what will have been implemented against the design and here are the follow up items that are necessary. The follow up items are sometimes follow ups to implement other aspects of the design and other times they are follow ups to the completion of the design itself, so things that are out of scope of the design, but which the design subsequently requires of the rest of the system. The realization that I’ve had is that in the wiki a design is not a single file, but in fact, a directory. Within the directory, I can have a target specification, and then separate from that I can capture other documents which support the design which speak to the phasing of implementation against the design. What I imagine that could look like in practice is you have the design directory within which is a file with a standard name like target spec or design.MD and there’s a nested sub directory called phases and the phases directory has at least a phase 1.MD implementation which serves the role of capturing them first implementation scope, which is often the most necessary to understand in design iterations and then there is a later.MD file containing all of the follow up items, not included in phase one and potentially the follow up items required of the design itself. On second thought, I think those two concerns should be split as well, so either part of the design or as a separate non-phased document we capture follow ups that must be implemented following the completion of one or more or all phases of the design itself. The purpose of calling it later as opposed to a phase 2 is that it’s just a grab bag of all of the things that we need to remember and at the current point in the process, it’s not required or necessary to identify the scope of subsequent phases or roadmap all of that out. That’s not a requirement, but nothing should prevent the user from doing that phasing if so desired. No thinking about a situation where a design does not need to be phased and the implementation happens in one go. I think it’s simpler to stick with that pattern and simply express in phase. One that phase one is to implement the design to its fullest. That obviates any questions about if there are no phased documents because there are no phases or are there no phase documents because we forgot. if we’re turning the design document into a directory, I think therefore it’s useful to be able to rely on standard file names so that any consumer of that knows where to look to find the information it’s looking for like the design file or the later file or the phase one file. Again that does not require that those are the only files in there, but at least the core structure follows a standard format.
